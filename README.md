# スロット双六 — Firebase連携版 セットアップ手順

## 1. ファイル構成

```
slot-sugoroku/
├── index.html          … ゲーム本体（既存UI・アニメーションは変更なし）
├── admin.html           … 管理画面（別ページ）
├── aikotoba.html         … あいことば専用ページ（別リンク。ゲーム画面からはリンクしない）
├── css/style.css
├── firestore.rules      … Firestoreセキュリティルール（そのままデプロイ可）
├── js/
│   ├── firebase-config.js  … ★要編集：あなたのFirebaseプロジェクト情報
│   ├── firebase.js         … Firebase初期化・共通export
│   ├── auth.js             … 匿名ログイン／管理者ログイン
│   ├── profile.js          … プロフィール（ユーザーネーム・称号・ベストスコア）
│   ├── ranking.js          … ランキング（playersを直接クエリ）
│   ├── achievement.js      … 実績定義・解除判定
│   ├── title.js            … 称号定義
│   ├── events.js           … イベントマス・Lucky Meter
│   ├── password.js         … あいことば（引き換え処理＋管理者CRUD）
│   ├── aikotoba-page.js     … あいことば専用ページ(aikotoba.html)のロジック
│   ├── admin.js             … 管理画面ロジック
│   ├── statistics.js       … 統計（プレイヤー別／グローバル）
│   ├── game.js              … ゲーム本体ロジック（Firebase非依存）
│   ├── ui.js                … モーダル・トースト等の汎用UI
│   ├── effects.js           … 演出（既存＋新規のLucky/Trap/実績演出）
│   ├── sound.js             … 効果音（WebAudio、外部音源不使用）
│   └── main.js               … エントリーポイント（配線役）
```

## 2. Firebaseプロジェクトの準備

1. https://console.firebase.google.com/ でプロジェクトを作成
2. Authentication → Sign-in method で **匿名** を有効化（プレイヤー用）
3. 同じく **メール/パスワード** を有効化（管理者ログイン用）
4. Firestore Database を作成（本番モード）
5. `firestore.rules` の内容をそのままFirebaseコンソールのルールに貼り付けてデプロイ
6. プロジェクト設定 → 全般 → 「マイアプリ」でウェブアプリを追加し、
   表示された設定値を `js/firebase-config.js` に貼り付ける

## 3. 管理者アカウントの作成

1. Authentication で管理者用のメール/パスワードアカウントを1つ作成
2. 発行されたUIDをコピー
3. Firestoreに `admins/{そのUID}` というドキュメントを手動作成（中身は空でOK）
4. `admin.html` からそのメール/パスワードでログインすると管理画面が開きます

## 4. あいことばの登録・配布

`admin.html` の「あいことば管理」タブから、あいことば・配布コイン数・
利用期間・有効/無効・最大利用回数を設定できます。

### あいことばの入力は専用ページ（別リンク）から行います

秘匿性を高めるため、あいことばの入力欄は**ゲーム画面(index.html)には置いていません**。
代わりに `aikotoba.html` という独立したページを用意しています。

- ゲーム内のどこにもこのページへのリンクは貼っていません。SNSやキャンペーンなど、
  周知したい相手にだけ直接URLを共有してください。
- `aikotoba.html?code=あいことば` のように `?code=` を付けたURLを配布すると、
  開いた時点で入力欄に自動入力されます（送信は利用者がボタンを押す必要があります）。
- 検索エンジンに拾われないよう `<meta name="robots" content="noindex, nofollow">` を設定済みです。

### 仕組み（別ページとゲーム画面をどう繋いでいるか）

`aikotoba.html` はゲーム画面のコード（`game.js`/`main.js`）に一切依存しない、
完全に独立したページです。あいことば引き換えに成功すると、その場でゲームの
コインを増やすのではなく、Firestoreの `players/{uid}.pendingCoins`
（未受取コイン）に加算するだけを行います。

その後、本人がゲーム画面(`index.html`)を開くと：

1. 起動時、および以後はプロフィールのリアルタイム監視によって
   `pendingCoins` が0より大きいことを検知
2. Firestoreトランザクションで安全に「読み取って0にリセット」しつつ、
   その金額をローカルのゲームコインに加算
3. 「あいことばで獲得した◯◯コインを受け取りました」とトースト表示

同じ端末・同じブラウザ（＝同じ匿名ログインuid）であれば、別タブ・別タイミングで
開いても確実に反映されます。逆に言うと、`aikotoba.html` は必ず**ゲームをプレイしている
のと同じ端末・ブラウザ**で開いてもらう必要があります（別の端末では別のuidになるため）。

## 5. ローカルで動かす際の注意

このアプリは ES Modules（`import`/`export`）を使用しているため、
`index.html` を **直接ダブルクリックでは動作しません**（ブラウザのCORS制限）。
以下のいずれかの方法でHTTPサーバー経由で開いてください。

```bash
# 例：Node.jsがある場合
npx serve .

# 例：Pythonがある場合
python3 -m http.server 8080
```

本番運用時は Firebase Hosting へのデプロイを推奨します。

## 変更点・仕様上の判断（元のご依頼との差分）

ご依頼の「テキストは必要があれば変更してよい」「変更した場合は理由を説明する」
という指示に従い、以下の点は具体的な数値・仕様が明記されていなかったため、
既存のゲームバランスを崩さない範囲で設計判断を行いました。

| 項目 | 判断内容 | 理由 |
|---|---|---|
| イベントマスの出現率・報酬額 | 通常65% / Lucky15%(+報酬2倍) / Trap12%(-10〜20コイン) / Bonus8%(+50〜80) | 具体的な確率・金額の指定がなかったため、既存の+5〜+30の報酬感を大きく壊さない範囲で設定。`events.js`内の定数を変更するだけで調整可能な構成にしています |
| Trapの効果 | コインのみ減少させ、スコアは減らさない | 「スコア＝獲得コインの累計」という既存仕様を尊重し、Trapを「一時的な所持金の目減り」として実績・ランキングには影響しない設計にしました |
| Lucky Meter発動タイミング | メーターがMAX(10)になった状態で迎えた次のスピン解決時に50%抽選、成功失敗に関わらずメーターを0にリセット | 「発動時50%／発動後0」という記述を、この解釈で実装しています |
| ランキングの実体 | 専用の`rankings`コレクションは作らず、`players`コレクションを`bestScore`降順でクエリ | ご依頼文に「ランキングはプロフィールを参照する設計」「ユーザーネーム変更が自動反映される」と明記されていたため、二重管理を避けるためにこの設計にしました |
| 実績の「定義」の保存場所 | Firestoreではなく`achievement.js`内の静的データ | 実績の判定には条件ロジック（しきい値比較）が伴うため、Firestore上のデータだけで完結させるとロジックの二重実装が必要になり保守性が下がると判断しました。実績の「解除状況」自体はFirestoreに保存し、要件の「実績をFirestoreへ保存する」は満たしています |
| 管理画面の実装場所 | `index.html`内のモーダルではなく`admin.html`という別ページ | 匿名ログイン（プレイヤー）とメール/パスワードログイン（管理者）はFirebase Authentication上で同時に2つのセッションを維持できないため、同一タブ内で混在させると挙動が不安定になります。ページを分けることで安全に両立させました |
| 総プレイ数のカウント方法 | ゲーム読み込み時・リセット時に+1 | 「プレイ」の定義が明記されていなかったため、1セッションの開始を1プレイとしました |
| 統計の「最高スコア」表示 | 別途保存せず、ランキング1位の`bestScore`を参照 | `players`側に既にベストスコアがあるため、統計用に同じ値を別ドキュメントへ複製すると不整合の原因になるため参照方式にしました |
| あいことば入力欄の設置場所 | ゲーム画面内モーダルを廃止し、`aikotoba.html`という独立ページに変更 | ご要望「別リンクから行えるようにすれば秘匿性が上がる」に対応。ゲーム画面のコードにあいことば入力UIが存在しなければ、一般プレイヤーが偶然発見する可能性を下げられます |
| 別ページで引き換えたコインの受け渡し方法 | `players/{uid}.pendingCoins`（未受取コイン）に加算し、ゲーム画面が起動時／リアルタイム監視で自動回収 | あいことば引き換えとゲーム画面が別ページ（別タブになりうる）のため、コインの受け渡しにFirestoreを経由する必要があり、この方式にしました |

## セキュリティ上の既知の制限

- **あいことばの正誤判定をクライアント側のFirestore読み取りで行っています。**
  Cloud Functionsを使わない構成のため、技術的に詳しい利用者がFirestoreの
  通信内容からコードを推測できる可能性があります。厳密な運用が必要な場合は
  あいことば検証をCloud Functions化することを推奨します（`firestore.rules`にも
  同様の注記を記載しています）。
- `statistics/global` への書き込みはログイン済みユーザーであれば誰でも
  実行できるルールにしています（クライアントのみ構成のための制約）。
  不正な値の書き込みを厳密に防ぎたい場合はCloud Functions経由に変更してください。
- `passwords/{id}.currentUses` は、あいことば引き換え時にプレイヤー自身が
  「+1だけ」更新できるようルールで許可しています（それ以外のフィールド変更は管理者のみ）。
  理論上、Firestoreへ直接不正なリクエストを送れば`currentUses`を水増しされる余地は残りますが、
  実害は「利用回数上限が早く切れる」程度に限定されるよう設計しています。
- `players/{uid}.pendingCoins` はあいことば引き換え時に加算されますが、ルール上は
  「本人が自分のドキュメントを更新する」操作として扱われるため、技術的には本人が
  Firestoreへ直接不正な値を書き込める余地が残ります。厳密に不正加算を防ぎたい場合は
  Cloud Functions経由でのみpendingCoinsを更新できるよう変更してください。

## 拡張しやすさについて

- 実績を増やす → `achievement.js`の配列に1行追加
- 称号を増やす → `title.js`の配列に1行追加
- イベントマスの種類・確率を変えたい → `events.js`の定数を変更
- ゲーム設定をFirestoreから動的に読み込みたい → `gameSettings`コレクションを
  読み取る処理を`game.js`の`CONFIG`初期化部に追加するだけで対応可能な構成にしています
